# 详细开发计划

本文把 NextBuf 从项目初始化到长期平台化拆成按功能交付的版本。它是产品、开发、测试和发布共同使用的执行基线，不是不可调整的日期承诺。

## 1. 版本规划原则

### 1.1 为什么项目骨架不叫 v1.0.0

`v1.0.0` 按语义化版本应代表首个可以公开部署、具有升级承诺的稳定版本。如果把只有项目骨架的代码发布为 `v1.0.0`，后续注册登录、发帖等正常功能就只能错误地塞进补丁版本，版本号会失去意义。

因此采用以下规则：

- `v0.x.0`：正式版之前的功能里程碑，可以发生较大内部变化。
- `v1.0.0-alpha.N`：核心链路尚未完整，只供开发测试。
- `v1.0.0-beta.N`：功能基本冻结，集中修复和验证升级。
- `v1.0.0-rc.N`：发布候选，只接受阻断发布的问题修复。
- `v1.0.0`：首个可运营、可升级、可恢复的稳定社区版本。
- `v1.0.1`：兼容性错误、安全或文档修复，不加入大型新功能。
- `v1.1.0`：向后兼容的新功能。
- `v2.0.0`：存在明确破坏性变更或进入新的平台能力阶段。

开发里程碑可以发布镜像，但 `v0.x` 必须标注为预览版，不承诺无损升级到所有后续预览版本。进入 Beta 前必须提供到 `v1.0.0` 的迁移路径。

### 1.2 版本不是分支

每个版本是一组可验收能力，不为每个长期保留独立开发分支。默认流程：

```text
Issue/Epic -> 功能分支 -> Pull Request -> main -> 版本标签
```

未完成的大功能使用功能开关隐藏，不能让主分支长期不可运行。

### 1.3 版本状态

- **Committed**：已进入近期开发范围，功能和验收标准应明确。
- **Planned**：方向明确，细节在进入开发前再次评审。
- **Exploratory**：长期能力地图，不构成发布时间或兼容承诺。

`v0.1.0` 至 `v1.0.0` 为 Committed 规划；`v1.1.0` 至 `v3.x` 为 Planned；`v4.x` 至 `v10.x` 为 Exploratory。

## 3. V1 前置开发：从空仓库到稳定版

### v0.1.0：项目骨架

状态：**Completed（2026-07-12）**

目标：建立所有后续功能共同依赖的仓库、构建和代码质量基础。

开发内容：

- 初始化 Next.js 16.2.10、TypeScript 严格模式、App Router 和 pnpm lockfile。
- 建立 `src/app/`、`src/components/`、`src/modules/`、`src/infrastructure/`、`src/worker/`、`tests/`、`deploy/` 目录。
- 配置 `server-only`、`*.server.ts` 和导入边界规则，阻止 Client Component 引入数据库、Redis、队列或密钥代码。
- 配置 ESLint、Prettier、TypeScript 检查和统一导入别名。
- 建立环境变量 Schema、开发/测试/生产配置入口。
- 建立结构化日志、请求 ID、统一错误类型和错误响应格式。
- 建立基础 CI：安装、格式检查、lint、类型检查、单元测试、构建。
- 添加标准 AGPLv3 `LICENSE`、第 7(b) 条附加署名 `NOTICE`、许可证说明、贡献指南、Issue 和 Pull Request 模板。
- 建立版本信息页面或 CLI，使运行实例能报告版本、提交和构建时间。

不包含：数据库业务模型、注册、发帖和正式页面。

验收门槛：

- 新环境按照 README 可以完成安装和 `pnpm dev`。
- `pnpm build`、类型检查和测试在 Node.js 24 LTS 下通过。
- CI 对主分支和 Pull Request 生效。
- 空首页、错误页和健康检查入口可以访问。
- `package.json`、LICENSE、NOTICE、README 和贡献条款对许可证的描述一致。

- 建立 Outbox 表、Dispatcher 骨架和幂等任务辅助工具。
- 实现 `/health/live`、`/health/ready` 及 Worker 健康状态。
- 提供开发环境 Compose，验证 PostgreSQL 18 数据卷路径。
- 建立测试数据库隔离、迁移测试和 Redis 集成测试工具。

验收门槛：

- 空库能通过 `setup` 初始化，重复执行不会破坏数据。
- Web 可写入一条测试事务，Outbox 可投递到 Worker 并完成幂等消费。
- 单独停止 Worker 不影响 Web 健康；恢复后能继续处理任务。
- 清空 Redis 后业务事实数据仍完整，缓存可以重建。

### v0.4.0：注册、登录和账号安全

状态：**Completed（2026-07-12）**

目标：建立可用于生产的身份认证主链路。

开发内容：

- 采用 Better Auth 1.6.23 并记录 ADR，完成账号、凭证、会话和验证令牌模型。
- 实现邮箱密码注册、邮箱验证、登录和退出。
- 实现忘记密码、重置密码和全部设备退出。
- 实现安全 Cookie、会话过期、撤销和设备列表。
- 实现登录、注册、验证、重置的限流和防账号枚举。
- 建立 OAuth Provider 接口，首个 Provider 根据决策启用。
- 实现注册策略：开放、邀请制、关闭注册。
- 记录关键身份安全审计事件。
- 使用加密 PostgreSQL Outbox、独立 Worker 和 SMTP 发送身份邮件，开发/测试使用 Mailpit。

验收门槛：

- 注册到邮箱验证再到登录的 E2E 通过。
- 重置密码后旧会话按策略失效。
- 未登录用户无法访问受保护接口，前端隐藏不是唯一防线。
- Token 数据库只保存哈希，日志不出现密码、Cookie 或验证码。
- 重复注册响应不泄露账号是否存在；邀请次数由 PostgreSQL 原子条件更新。
- 本阶段只创建真实名称、邮箱、账号和会话，不创建 UID、`@username`、资料或信任等级。

### v0.5.0：用户资料与账号中心

状态：**Completed（2026-07-12）**

目标：完整实现 UID、用户名、昵称和公开用户页面。

开发内容：

- 使用独立 PostgreSQL 序列从 1 起分配不可变且不复用的数字 UID，并为既有用户回填；早期 Beta 已分配 UID 不重新编号。
- 实现 3-24 位小写 ASCII `@username`、可重复昵称、服务端规范化和数据库唯一约束。
- 实现用户名保留词、30 天修改冷却期和永久历史别名跳转。
- 实现浏览器 512px WebP 居中裁剪/缩放、服务端大小与文件签名校验，以及可替换的本地头像存储边界。
- 实现简介、主页、注册时间、活动统计等公开资料。
- 实现账号中心：资料、安全、会话、通知偏好占位和隐私设置。
- 实现公开用户页、头像下拉菜单和用户名跳转。
- 实现 14 天可撤销且重复请求不延期的注销申请；最终匿名化和清理留到隐私政策确定后。
- 仅显示初始 `TL0`，不提前实现 `v0.10.0` 的持久化信任计算或授权能力。

验收门槛：

- UID 不可修改、不复用；用户名规范化后有数据库唯一约束。
- 用户名变更后旧链接按策略跳转，不能立即被冒用。
- 头像上传经过权限、同源、大小和文件签名校验，替换后旧本地文件被清理。
- 用户菜单按设计显示昵称、`@username`、UID 和 TL，不显示专业标签。
- 未激活账号没有公开资料页；邮箱不在公开用户页显示。
- Profile 由迁移回填并由数据库触发器覆盖所有用户创建入口。

### v0.6.0：节点与主题

状态：**Completed（2026-07-12）**

目标：让用户可以浏览节点并发布、查看、编辑主题。

开发内容：

- 实现节点目录、节点详情、排序读取、归档/可见性字段，以及仅管理员可调用的创建/更新用例和 Route Handler。
- 实现主题创建、草稿、客户端纯文本预览、发布、编辑、软删除、恢复和“我的主题”。
- 用 PostgreSQL 查询替换首页演示节点/主题，支持首页与节点主题流、最新/精华/计算型热门筛选和稳定双向游标。
- 实现置顶、精华、关闭、隐藏状态；热门由 30 天活动窗口和回复/浏览阈值计算，不保存永久布尔值。
- 发布规则为标题 6-120 字、正文 20-20000 字、最多 5 个 HTTP(S) 链接、每小时最多 3 个主题；草稿每人最多 20 个。
- 实现作者、节点版主、全局版主、管理员服务端授权骨架；他人私有草稿不向版主公开。
- 每次内容版本进入不可变修订表，发布、编辑、管理、删除、恢复和节点变更进入事务内社区审计。

验收门槛：

- 每个已发布主题必须属于有效节点。
- 主题列表符合静态稿：节点在第二行灰色标识，状态位于标题右侧，不显示标签。
- 普通用户不能修改他人主题或管理状态。
- 删除与恢复不会破坏主题 ID、修订和审计关系。
- 普通回复、Markdown、附件、点赞、收藏和通知未提前实现。

- 实现账号时长、阅读、发帖、反馈和违规等计算指标。
- 实现信任重算 Worker、批处理进度和影响预估。
- 保证管理角色、TL、专业声誉和交易信用独立。

验收门槛：

- 每个管理动作有操作者、原因、目标、前后状态和请求 ID。
- TL3 不自动成为版主，管理员权限不依赖 TL。
- 制裁立即影响服务端授权，不能仅靠缓存或 UI。
- 信任规则调整可分批重算并能解释单个用户的等级来源。

完成记录：

- 新增 Report、ModerationCase、ModerationAction、Sanction、GovernanceAuditEvent、TrustRuleVersion、TrustUserState、TrustLevelHistory 和 TrustRecalculationBatch，并由 PostgreSQL CHECK、外键、活动键唯一约束、默认规则和用户触发器兜底。
- 公开主题、回复和用户举报按目标聚合；同一举报者/目标只允许一条未结举报，每用户 UTC 日最多 10 条，举报快照保留当时 TL 与权重。

- 规则必须先完成 preview 批次才能激活；激活创建 apply 批次，Worker 以 UID 游标和事务性 Outbox 每批 25 人处理，并保存进度、影响分布、解释和历史。

- 后台采用工作流服务而不是任意表 CRUD；普通用户直接调用后台 Provider API 的 Playwright 旅程必须返回 403。

## 14. 每个版本的统一开发流程

### 14.1 版本启动

1. 确认版本目标和不做事项。
2. 将功能拆为 Epic、用户故事和技术任务。
3. 确认数据库、权限、API、Worker 和部署影响。
4. 为高风险决策创建 ADR。
5. 定义迁移、回滚和观测指标。

### 14.2 开发顺序

单个功能按以下顺序推进：

```text
领域规则 -> 数据约束/迁移 -> 应用服务 -> 权限
-> Worker/事件 -> API/Server Action -> UI -> 测试 -> 文档
```

可以并行开发，但不能在领域和授权尚未定义时只完成页面外观后宣称功能完成。

### 14.3 发布候选

- 功能冻结，只接受该版本范围内的修复。
- 在旧版本数据副本上执行迁移。
- 使用最终镜像和 Compose 完成安装、升级、备份、恢复。
- 执行安全、性能、E2E 和多架构冒烟测试。
- 更新变更日志、已知问题和回滚限制。

### 14.4 发布后

- 观察错误率、数据库、队列、邮件和关键业务指标。
- 对严重问题发布补丁，不在紧急补丁中夹带新功能。
- 记录实际迁移时间、资源消耗和用户反馈。
- 将遗留问题进入下一版本，不静默扩大当前版本范围。
- 推送 `main` 并确认远程 SHA 与本地一致。
- 为完成的里程碑创建并推送注释标签，例如 `v0.1.0`。
- 确认远程 CI 通过后，才把该版本视为正式回滚点。

## 15. 单个版本的完成定义

每个版本发布前必须满足：

- 所有标记为 Must 的功能完成，Should 项可以明确延期。
- 数据库迁移从受支持前一版本验证成功。
- 权限拒绝路径与正常路径都有测试。
- Worker 任务具备幂等、重试和失败处理。
- 桌面、移动和键盘操作完成检查。
- 部署配置、环境变量、备份和升级文档同步更新。
- 无未说明的严重安全、数据损坏或升级阻断问题。
- 版本标签、镜像、源码、SBOM 和发布说明相互对应。

## 16. 如何调整本计划

可以调整版本顺序，但必须遵守依赖关系。例如：

- 用户资料依赖身份系统，不能先公开用户页再补权限。
- 通知依赖主题、回复和 Outbox，不能只做前端红点。
- 管理后台依赖治理用例，不能先生成全表 CRUD。
- 插件依赖稳定事件和 Provider，不能在核心模型仍频繁变化时冻结 API。
- 支付依赖订单、账务、幂等和法务评审，不能从一个支付按钮开始。

调整步骤：

1. 在本文件修改版本范围和依赖。
2. 在决策台账记录重大变化。
3. 若影响架构、数据库或部署，新增或替代 ADR。
4. 更新 Issue/Epic，不让文档与实际开发计划长期分叉。
