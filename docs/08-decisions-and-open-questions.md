# 决策与待讨论事项

本文件是当前决策台账。它记录“为什么”，避免未来只看到结果却不知道约束。

## 1. 已确定决策

| ID | 决策 | 原因 | 主要影响 |
| --- | --- | --- | --- |
| D-001 | 使用 Next.js 16.2.10、TypeScript、App Router | 前后台同构、SSR 和单仓交付符合产品目标 | 锁定 Node 服务端运行方式 |
| D-002 | 使用 Node.js 24 LTS | 支持周期长，统一 CI、镜像和非 Docker 环境 | 不支持旧 Node 环境 |
| D-003 | 前台、后台、API 和 Worker 位于同一仓库 | 共享领域规则和发布版本，降低早期分布式复杂度 | 需要严格模块边界 |
| D-004 | 采用模块化单体，不采用微服务 | V1 团队和业务尚不需要跨服务一致性成本 | 内部模块通过代码接口和事件协作 |
| D-005 | PostgreSQL 18 是唯一官方数据库 | 社区查询、全文搜索、约束和后续扩展更适合 PostgreSQL | 不承诺 MySQL 兼容 |
| D-006 | Redis 8 + BullMQ | 支持缓存、限流、可靠队列和独立 Worker | Redis 成为运行依赖但不是事实来源 |
| D-007 | 构建一个应用镜像 | 简化版本、发布和用户理解 | 镜像同时包含 Web、Worker、迁移入口 |
| D-008 | 默认四个常驻容器 | Web/Worker 故障隔离，数据库和 Redis 独立持久化 | Web、Worker、PostgreSQL、Redis |
| D-009 | setup/migrate 是一次性任务 | 避免多个 Web 实例争抢迁移 | Compose 需等待 setup 成功 |
| D-010 | 权限角色、信任等级、专业声誉、交易信用分离 | 四者来源、风险和生命周期不同 | UI 和数据模型不得合成一个等级 |
| D-011 | V1 不运行任意第三方服务端插件 | 插件安全、迁移和兼容成本过高 | 先保留事件和 Provider 扩展点 |
| D-012 | 同时支持 Docker 与非 Docker 部署 | 覆盖宝塔用户和传统 Linux 运维 | 发布需包含镜像与 standalone 包 |
| D-013 | 稳定版之前使用 v0.x 功能里程碑，v1.0.0 代表首个稳定版 | 保持 SemVer 含义，避免用补丁版本承载大型功能 | v1.0.x 只修复，新增兼容功能进入 v1.x.0 |
| D-014 | 所有应用源码进入 `src`，不拆顶层 frontend/backend | Next.js 同时承载页面和服务端，物理拆分会破坏共享运行时优势 | 使用 `src/app`、`components`、`modules`、`infrastructure`、`worker` 分层 |
| D-015 | 环境变量以配置参考为公开合同 | Web、Worker、CLI、Compose 必须使用相同语义，避免部署漂移 | v1.0.0 后变量改名需要兼容与弃用周期 |
| D-016 | 单机发布提供 `nextbufctl` 薄封装 | 降低宝塔/命令行用户操作门槛，同时保留标准 Compose 可见性 | 脚本只编排公开命令，不隐藏另一套部署逻辑 |
| D-017 | V1 使用 Topic + 统一 Post 模型 | 首帖和回复共享编辑、修订、删除、审核和插件语义 | Topic 保存元数据，position=1 的 Post 是首帖 |
| D-018 | V1 只提供单一“点赞”反应 | 降低数据、信任计算和 UI 复杂度 | 多反应类型推迟到 V1.x 评估 |
| D-019 | V1 不提供用户标签系统 | 节点已承担主要分类，首页也明确不展示标签 | 后续按搜索和运营需求评估管理员标签/用户标签 |
| D-020 | V1 不提供用户私信 | 私信会显著增加骚扰、举报、隐私和存储范围 | 后续版本单独立项，不混入通知模块 |
| D-021 | 核心采用 AGPL-3.0-only，并依据第 7(b) 条要求保留页脚署名链接 | 防止网络托管修改版闭源，同时保留原项目来源 | 所有 Web 版本必须显示 `Powered by NextBuf` 官方链接，不增加非商业限制 |
| D-022 | 外部贡献采用 DCO 1.1，不使用 CLA | 项目保持 AGPL 开源且暂无闭源双许可计划，DCO 更轻量透明 | 每个贡献提交必须包含 `Signed-off-by`，未来整体再许可需取得贡献者同意 |
| D-023 | 使用 Zod 作为统一运行时输入校验工具 | v0.1.0 环境 Schema 验证通过，适合 Web、Worker 和 CLI 共享 | 环境、表单和 API 在边界解析，不信任 TypeScript 静态类型 |
| D-024 | 单元测试使用 Vitest | 与 TypeScript、Node.js 24 和当前构建兼容，v0.1.0 已验证 | 数据库集成测试仍使用真实 PostgreSQL/Redis，不能只做 mock |
| D-025 | 每个完成阶段必须推送并创建版本标签 | 远程提交和标签提供稳定回滚点，避免故障时依赖某台开发机 | 检查通过后更新文档、sign-off 提交、推送 main、创建并推送注释标签 |

### 贡献授权方式

核心许可证和贡献授权方式已经确定，详细规则见 [许可证与署名政策](./15-license-attribution-policy.md)：

- 使用 DCO 1.1 和 `Signed-off-by`，贡献者保留版权并确认有权按项目许可证提交代码。
- 不要求 CLA，也不要求贡献者转让版权。

该选择意味着项目所有者以后不能当然地把所有外部贡献改成闭源许可证；重大再许可需要取得相关贡献者同意。

### 对应后续大版本前

- 交易适用地区、支付 Provider、退款与争议责任。
- 专业认证由谁签发，如何撤销和申诉。
- 插件的信任模型、签名、数据库迁移和服务端权限。
- 多节点究竟指横向扩展、地域容灾还是社区联邦。
- 是否建立独立搜索服务和观测平台的容量阈值。

## 4. 决策记录流程

出现以下变化时，应新增 ADR，而不是只改表格：

- 更换数据库、ORM、认证库或队列。
- 新增常驻基础服务。
- 改变单镜像或四容器部署模型。
- 暴露稳定插件 API 或公开 API。
- 改变用户身份、权限或内容所有权语义。
- 引入不可逆数据库迁移或破坏兼容的配置变更。

ADR 文件放在 `docs/adr/`，命名示例：

```text
0001-use-postgresql.md
0002-separate-web-and-worker-runtime.md
```

每份 ADR 至少包含：状态、日期、背景、决策、备选方案、正面影响、负面影响、迁移和回退方式。

## 5. 变更原则

“已确定”不代表永远不能改变。它表示改变前必须：

1. 提供当前方案无法满足的具体证据。
2. 比较至少一个现实备选方案。
3. 说明数据、API、部署和用户迁移影响。
4. 更新对应文档、测试和发布说明。
5. 对不可逆变化提供明确审批和恢复计划。
