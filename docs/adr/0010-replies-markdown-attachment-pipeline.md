# ADR-0010：回复、Markdown 与附件管线

- 状态：Accepted
- 日期：2026-07-13

## 背景

`v0.7.0` 需要在 ADR-0005 的统一 Topic + Post 模型上开放普通回复，并同时解决并发楼层、编辑修订、引用、Markdown 安全、附件异步处理和文件回收。楼层与历史修订是长期内容合同；附件又跨越 Web、PostgreSQL、Redis/Outbox、Worker 和对象存储，不能只靠客户端状态或正文字符串决定是否删除。

## 决策

### 稳定楼层与回复事务

- 首帖继续固定为 `position=1`；Topic 保存从 2 开始的 `next_post_position`。
- 创建回复时先锁定作者行执行每小时 20 条的 PostgreSQL 事实限流，再锁定 Topic 行读取和递增楼层计数器。
- 回复、version=1 修订、提及、附件引用、Topic 有效回复数/最后活动时间和审计事件在同一事务提交。
- 同一 Topic 的 `(topic_id, position)` 保持数据库唯一；删除回复只改状态并保留楼层，恢复不会分配新楼层。
- 引用保存目标 Post ID，数据库触发器拒绝跨 Topic 引用。主题关闭后普通成员不能回复，具有该节点管理能力的角色可以例外回复。

### Markdown 与提及

- `body_source` 中的 Markdown 是事实来源；预览和正式页面都调用同一个服务端渲染器。
- 使用 remark/rehype 管线解析 GFM，禁用原始 HTML，并在生成 HTML 后执行允许列表清洗。
- `javascript:` 等危险协议被移除；站外 HTTP(S) 链接使用 `nofollow noopener noreferrer` 并在新窗口打开。
- 站外图片不直接嵌入，只显示为安全链接；只有 NextBuf 附件媒体路由可以生成 `<img>`。
- `@username` 在非链接、非代码文本中解析为用户页链接；只为当前有效用户名和 active 用户写入去重后的 Mention 事实。本版本不创建通知。
- 本版本不抓取远程图片或链接预览，因此没有服务端远程请求入口。未来若增加抓取，必须在请求时同时校验协议、DNS 解析结果、重定向链和私有/本地地址，不能只复用字符串检查。

### 附件存储与处理

- 头像和通用附件共用 `local`/`s3` 对象存储边界；Attachment 记录保存实际写入时的 driver 和不可变随机对象键。
- Web 按文件签名识别 PNG、JPEG、WebP、PDF、UTF-8 文本和 ZIP，并校验声明 MIME、字节上限、active 账号和每小时 20 个上传的 PostgreSQL 行锁限流。
- 原始对象始终保留。图片由事务性 Outbox 投递给 Worker，Worker 使用 Sharp 解码、限制像素并生成去元数据的 WebP 派生文件；失败状态和错误原因保存在 PostgreSQL。
- 当前 Post、不可变 Revision 和回复/主题 Draft 分别使用关系表引用附件。内容事务在建立或移除引用时锁定 Attachment 行，延迟回收任务使用同一行锁，避免新引用与删除竞争。
- 无任何引用的附件记录 `orphaned_at`，经过默认 24 小时宽限期后由 Worker 再次检查引用并删除原始/派生对象和数据库记录。
- ready 且被公开 Topic 中 published Post 引用的附件可以匿名读取；未引用、pending 或 failed 附件只允许上传者读取原始对象，响应使用私有无缓存策略。

## 备选方案

### 通过 `MAX(position) + 1` 分配楼层

并发事务会竞争并产生唯一冲突或重试风暴，删除后也容易错误复用楼层。Topic 单调计数器使分配语义明确，并由行锁和唯一约束共同保证。

### 只保存渲染后的 HTML

渲染规则升级后无法安全重建，编辑和迁移也会丢失作者源内容。保存 Markdown 源并在服务端渲染更适合长期兼容。

### 仅从当前正文字符串判断附件是否在用

这会在编辑、恢复草稿或查看历史修订时丢失文件。显式的当前/修订/草稿引用关系可以由外键和事务保护。

### Web 请求内同步处理图片

大图解码会增加请求延迟和内存峰值，失败重试也缺少可追踪状态。Worker 派生允许独立并发、重试和诊断，同时保留原始对象。

### `v0.7.0` 只支持本地文件

单机更简单，但会继续阻塞多 Web 实例和对象存储部署。统一 Provider 在本版本同时实现 local/S3，避免未来改变 Attachment 数据合同。

## 后果

- 回复写入会串行锁定单个 Topic；这是稳定楼层的有意取舍。高流量 Topic 的分片或独立分配器只能在有容量证据后另建 ADR。
- PostgreSQL 是楼层、计数、提及、处理状态和引用事实来源；Redis 丢失不能改变已发布内容。
- Worker 现在同时负责邮件与附件处理，部署时必须持续运行并为 Sharp 图片处理预留内存。
- 本地存储目录成为必须备份的事实数据；多 Web 实例必须使用 S3 或经过验证的共享文件系统。
- 已公开且采用长期缓存的附件不能承诺从第三方缓存中即时撤回，敏感文件不应作为公开帖子附件发布。

## 迁移与回退

迁移为既有 Topic 增加 `next_post_position=2`，因为 `v0.6.0` 尚无普通回复；同时新增回复草稿、提及、附件及三类引用表，并让回复修订的标题可空。迁移只增加结构，不删除既有主题数据。

数据库包含 `v0.7.0` 回复后，不应删除新表或重置楼层计数器。短期代码回退到 `v0.6.0` 会让回复界面不可用但可保留新增数据；正式恢复应发布向前修复。若必须完全回退，只能恢复升级前 PostgreSQL 与附件备份，并接受升级后内容丢失。
