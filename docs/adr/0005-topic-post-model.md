# ADR-0005：主题首帖与回复统一使用 Post

- 状态：Accepted
- 日期：2026-07-12

## 背景

社区主题包含标题、节点、置顶、关闭、热门等主题级元数据，同时包含首帖和后续回复。可以把首帖正文直接存在 Topic、回复存在 Reply，也可以让首帖和回复统一进入 Post。

首帖与回复都需要 Markdown 渲染、编辑历史、软删除、引用、附件、举报、审核和未来插件扩展。分开存储会让这些能力维护两套代码和查询语义。

## 决策

- `Topic` 保存节点、标题、作者、状态、管理标志、统计和最后活动等主题级元数据。
- `Post` 保存首帖与回复的内容、作者、position、状态和时间。
- 每个 Topic 的 `position = 1` Post 是首帖。
- 后续回复使用稳定递增 position，软删除不重排。
- Topic 与首帖在同一个数据库事务中创建。
- `(topic_id, position)` 建立唯一约束。
- 编辑内容写入统一 Revision 机制。

Topic 可以保存 `first_post_id` 作为显式引用，也可以通过唯一的 position=1 查询；最终 Prisma Schema 选择需避免无法初始化的循环非空外键，但不能改变统一 Post 语义。

## 备选方案

### Topic 保存首帖，Reply 保存回复

读取首帖直接，但编辑、修订、附件、举报和审核要同时支持两种实体，长期插件 API 也更复杂。

### 所有信息都保存到 Post

主题标题、节点、关闭和置顶不属于单个内容楼层，强行放进 Post 会让主题列表和管理查询变得不自然。

## 后果

正面影响：

- 首帖和回复共享内容处理、修订、审核、引用和附件逻辑。
- 插件和公开 API 可以使用统一内容标识。
- 软删除和楼层关系更容易保持稳定。

负面影响：

- 创建主题需要同时写 Topic 与首帖。
- 首页列表需要关联首帖摘要时产生 join 或摘要缓存。
- 必须正确处理楼层分配的并发唯一性。

## 实现约束

- 楼层 position 在事务内分配，不能先读最大值后无保护地加一。
- Topic 中的回复数、最后回复时间等是可重建聚合，不取代 Post 事实。
- 删除首帖通常表现为删除/隐藏整个 Topic，不能留下没有首帖的可见主题。
- 主题作者通常与首帖作者一致；若保留两个字段，必须用事务维持一致并说明冗余目的。
