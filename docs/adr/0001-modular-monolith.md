# ADR-0001：采用 Next.js 模块化单体

- 状态：Accepted
- 日期：2026-07-12

## 背景

NextBuf 需要在同一产品中提供社区前台、管理后台、API 和异步任务。V1 团队规模、领域边界和流量尚不需要微服务，但项目后期会加入交易、开放 API 和插件，因此也不能把全部业务规则堆在页面和数据库调用中。

## 决策

使用 Next.js 16.2.10、TypeScript 和 App Router 构建模块化单体：

- 前台、后台、API、领域模块和 Worker 位于同一仓库和版本。
- 领域按 identity、community、moderation、trust 等模块划分。
- 模块通过公开应用服务或领域事件协作。
- Web 与 Worker 是运行角色，不是独立微服务。

## 备选方案

### 完全前后端分离

独立 Vue/React SPA 加 API 后端会增加两套认证边界、部署产物和类型合同，不符合前后台一起交付的目标。

### 从 V1 开始使用微服务

可以独立扩缩，但会立即引入服务发现、分布式追踪、跨服务事务和多版本 API 成本，当前没有足够收益。

### 无模块约束的单体

初期开发快，但管理、信任和社区内容会互相直接访问数据，后续插件和交易难以建立稳定边界。

## 后果

正面影响：

- 一次构建和发布即可更新完整产品。
- Web 与 Worker 共享类型、授权和领域规则。
- 数据库事务可以覆盖多数核心用例。
- 模块边界为后续 Provider、插件或必要的服务拆分提供基础。

负面影响：

- 必须通过代码审查和依赖规则维护模块边界。
- 单仓变更可能影响完整应用，需要可靠回归测试。
- 某个模块不能仅靠独立部署解决性能问题，需先优化或按运行角色扩容。

## 迁移与回退

这是初始架构，无现有数据迁移。未来拆分服务必须以具体容量或组织证据为依据，并为数据所有权、事件交付和 API 兼容另建 ADR。
