# ADR-0006：采用 AGPLv3 与页脚署名附加条款

- 状态：Accepted
- 日期：2026-07-12

## 背景

NextBuf 的目标是自用并建立长期开放的社区程序。项目允许他人安装、修改、收费部署和商业运营，但要求修改后的网络服务继续提供对应源码，并在页面底部保留原项目来源链接。

MIT、Apache 和普通 GPL 无法同时满足这两个目标：MIT/Apache 允许闭源修改，GPL 的源码义务通常由分发触发，单纯在线托管修改版可能不需要提供源码。标准许可证中的版权保留也不等于必须在 Web 页脚显示链接。

## 决策

- 核心使用 GNU Affero General Public License v3.0 only。
- 依据 AGPLv3 第 7(b) 条增加合理作者署名和法律通知要求。
- 所有提供 Web UI 的原始和修改版本必须在页面底部清晰显示 `Powered by NextBuf`。
- `NextBuf` 链接指向 NOTICE 中确定的官方项目仓库。
- 部署者可以增加自己的社区版权、品牌和链接，但不能删除、隐藏、替换或配置关闭 NextBuf 署名。
- 不增加禁止商业使用、仅供学习或未经许可不得收费等限制。
- 品牌和 Logo 的商标授权另行制定，软件许可证只授予满足署名所需的名称使用。

## 备选方案

### MIT/Apache-2.0

有利于采用和闭源集成，但允许竞争者修改后完全闭源运营，不符合项目目标。

### GPLv3

分发修改版时要求源码，但网络托管缺口不适合社区服务端程序。

### LGPLv3

适合允许闭源应用链接的库，不适合作为完整社区平台的主要保护方式。

### AGPLv3 无附加署名

可以保护网络修改源码，但不能把“全局页脚保留官方链接”的具体产品要求表达得足够明确。

### 非商业许可证

可以限制商业使用，但不符合 OSI 开源定义，也与项目希望成为真正开源项目的目标冲突。

## 后果

正面影响：

- 修改后的网络服务需要向用户提供对应源码。
- Fork 和二次部署仍保留 NextBuf 来源链接。
- 允许收费部署、托管服务和商业运营。
- 社区内容、数据库和部署秘密不因 AGPL 自动公开。

负面影响：

- 部分企业会因 AGPL 政策拒绝采用。
- 自定义第 7(b) 条需要准确法律文本和许可证扫描说明。
- 页脚实现、主题和插件系统必须长期保护署名组件。
- 未来整体再许可需要处理所有外部贡献版权，除非从开始采用合适 CLA。

## 实现与发布

- `v0.1.0` 添加 LICENSE、NOTICE、README 许可证说明和贡献政策。
- `v0.3.0` 在所有 Web 布局实现并测试署名。
- V3 主题和插件 API 禁止移除法律署名。
- 正式发布前确定官方仓库 URL、DCO/CLA，并完成附加英文条款法律复核。
