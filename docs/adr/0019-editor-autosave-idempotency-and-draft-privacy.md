# ADR-0019：编辑器自动保存幂等与私人草稿边界

- 状态：已接受
- 日期：2026-07-20
- 适用版本：`v0.13.8`

## 背景

公开 Beta 发现主题和回复编辑器存在一类数据正确性问题：1.5 秒自动保存、附件上传和显式发布可能在网络延迟下交错。浏览器取消请求或等待超时只代表客户端不再等待，不能证明服务端事务没有提交；如果发布后又落入一个迟到的自动保存，请求可能留下草稿、覆盖较新正文或让用户误以为内容没有发布。刷新、返回历史记录、重复点击和多个标签页会进一步放大这一问题。

同一轮审计还发现，后台列表、计数和审计查询不能因为管理员或版主拥有管理角色，就读取作者的私人草稿。删除自草稿的主题仍属于私人草稿谱系；历史数据若缺少 `deleted_from_status`，恢复语义会回到草稿，因此也必须按私人内容处理。

## 决策

### 1. 客户端只有一条写入队列

- 主题和回复编辑器各自使用严格串行的异步任务队列。前一个自动保存成功、失败或超时结算后，后一个写入才能开始。
- 显式保存或发布先用同步 ref 阻止重复点击，取消尚未触发的 debounce，等待队列空闲，再读取标题、节点、正文和引用的最新 ref 快照。
- 编辑请求最多等待 15 秒，附件上传最多等待 60 秒。超时通过 `AbortController` 停止客户端等待，但界面必须提示“无法确认是否已经提交”，不能宣称服务端失败。
- HTTP 2xx 只有在响应体可解析且符合预期结构时才算已确认成功；空响应、截断内容、无效 JSON 或结构不合法的 2xx 与超时一样属于“结果未知”，必须依靠编辑 key 恢复，不能直接重试成一次新的发布。
- 附件上传期间，主题和回复的发布按钮以及发布函数本身都拒绝提交。附件完成时以同步维护的正文 ref 插入 Markdown；即使用户切到预览，插入也不会丢失或覆盖最后输入。
- 组件卸载后，旧请求不得修改当前 URL、导航或更新组件状态。

### 2. 编辑会话绑定当前浏览历史项

- 浏览器为每次新主题和回复编辑尝试生成随机 UUID，并保存单调 revision；History state 只保存 UUID、revision 和作用域，不保存正文或密钥。
- 新主题额外使用当前 history entry 的随机 marker 与 URL hash 配对，防止一个固定存储槽把新的 `/topics/new` 错绑到旧草稿。
- 回复会话绑定 Topic 编号与当前 `pathname + search`；成功发布时在一次 history replace 中清除会话并切换到稳定楼层 URL。
- 刷新或返回历史项时，客户端通过仅限当前作者的恢复接口查询 PostgreSQL。已存在草稿时进入数字主题编辑 URL，已发布内容进入主题或楼层 URL；短暂 404 会有限次数重试，以覆盖原写入事务仍在提交的窗口。
- 恢复接口返回 `superseded` 时，客户端删除当前 history entry 中的旧回复会话并重新加载规范 Topic 页面状态；该终态不是需要持续重试的冲突，不能形成刷新恢复循环。
- 收到 `editor_session_conflict` 后当前编辑器冻结继续写入，要求刷新确认；客户端不能自行增加 revision 后覆盖另一个标签页已经接受的快照。

### 3. PostgreSQL 决定主题幂等

- `CommunityTopic` 保存 nullable `editor_session_key` 与 `editor_session_revision`；同一作者的非空 key 唯一。历史主题保持 `NULL/NULL`，不回填、不重编号。
- 新主题 POST 和主题内容 PATCH 必须同时提交 UUID 与 `1..2147483647` revision。旧缓存脚本和直接请求缺少字段时返回 400，不进入原来的非幂等路径。
- 主题内容写入统一按“用户行 -> Topic 行”加锁。同 key、同 revision、同快照是幂等重试；较低 revision 或同 revision 不同快照返回冲突；更高 revision 才能推进内容。
- 发布后的同 key 自动保存不再改正文；发布响应丢失后，即使主题随后关闭或作者被暂停，同 key 发布重试仍返回已经提交的终态。

### 4. 回复使用持久会话状态机

- `CommunityPostDraft` 和最终 `CommunityPost` 镜像保存编辑 key/revision，便于数据约束、诊断和恢复；同一作者的非空 key 分别唯一。
- `CommunityReplyEditorSession` 是回复会话的权威状态，保存 Topic、作者、key、revision、可选最终 Post 以及四种状态：
  - `active`：有当前草稿快照；
  - `cleared`：正文已明确清空，保留 revision 终点；
  - `published`：已绑定唯一最终 Post；
  - `superseded`：同一用户/Topic 的新会话已经接管，旧会话在逻辑上终止且正常客户端不得复用。
- 清空草稿会删除正文和附件引用，但不删除会话终点。迟到的低 revision 因而不能重新创建草稿。
- 不同 key 不能覆盖仍有正文的草稿。清空后新 key 可以接管，并把旧会话标为 `superseded`；发布事务把当前会话标为 `published`，删除可见草稿并绑定最终 Post。
- 同 key 发布重试在主题关闭、隐藏、作者受制裁或 Redis 丢失前，先返回已发布 Post。回复位置、修订、提及、附件引用、Topic 计数、审计、通知 Outbox 和会话终态仍在同一 PostgreSQL 事务提交。
- 每个用户在滚动一小时内最多创建 60 个新回复编辑会话；同 key 的自动保存、清空、发布或恢复重试不重复计数。达到上限后新 key 返回 `editor_session_rate_limited`，该限制在锁定用户行的 PostgreSQL 事务内执行，不能通过并发请求或清空 Redis 绕过。
- `cleared` 和 `superseded` 墓碑按 `updated_at` 至少保留 30 天，用于拒绝正常网络延迟、旧标签页和历史项中的迟到写入；Worker 周期维护每批最多删除 500 条已过期墓碑。`active` 与 `published` 会话不属于这项清理，正常客户端在看到 `superseded` 后必须立即清除旧 history key。
- 删除本人已有回复草稿是内容收回，不是创建新内容。即使 Topic 已关闭，或作者账号/制裁状态不再允许发帖，作者仍可提交空正文来删除已有 Draft 及附件引用；删除能力不能因发布权限丧失而失效，也不借此授予发布能力。站点 `repliesEnabled=false` 保持既有设置语义：只阻止正式发布，仍允许保存和编辑回复草稿。

### 5. 私人草稿谱系不属于后台内容面

以下 Topic 均为作者私人草稿谱系：

- `status=draft`；
- `status=deleted AND deleted_from_status=draft`；
- `status=deleted AND deleted_from_status IS NULL` 或未知值。

它们不进入后台主题/回复列表、搜索、状态筛选、仪表盘计数、活跃用户、用户内容数、节点主题数或社区审计查询/导出。管理员和版主不能通过主题详情或按数字编号调用内容修改、删除、恢复、管理接口读取其存在和正文；这些非作者请求统一返回与主题不存在相同的 `404 topic_not_found`。作者继续通过 `/account/topics` 管理。已发布、关闭、隐藏以及从这些公开状态删除的内容仍按既有角色作用域管理。

恢复删除主题时只允许 `published`、`closed`、`hidden` 三种公开来源恢复为公开状态；`draft`、`NULL` 或任何未知来源一律恢复为私人 `draft`，不能让异常历史值把私人谱系提升为公开内容。

## 数据库约束

- 三组 key/revision CHECK 显式要求成对 `NULL/NULL` 或成对非空且 revision 至少为 1，不能依赖 PostgreSQL CHECK 对 `NULL` 的宽松语义。
- Topic、Post、ReplyDraft 的作者/key 唯一索引允许多个历史 `NULL/NULL`，但拒绝同一作者复用非空 key。
- 回复会话 CHECK 限制 revision、状态以及 `published <-> post_id` 一致性；索引支持作者/key 恢复和 Topic/作者/状态查询。
- 回复会话索引还必须支持按作者/创建时间执行每小时新会话限制，以及按状态/更新时间扫描过期 `cleared`/`superseded` 墓碑。
- API 和服务层把 revision 上限固定为 PostgreSQL `INTEGER` 上限，超限输入在进入数据库前拒绝。

## 备选方案

### 只在按钮上禁用重复点击

拒绝。它不能处理 debounce、超时后仍提交的服务器事务、刷新、返回历史或多个标签页。

### 取消请求就当作服务端失败

拒绝。HTTP 连接中断与数据库事务是否提交没有原子关系；错误提示必须承认结果未知，并依赖幂等 key 恢复。

### 清空草稿时直接删除全部会话事实

拒绝。删除 revision 终点会让迟到保存重新创建草稿。正文可以删除，但终态必须保留并允许后续新会话显式接管。

### 永久保留全部 cleared/superseded 墓碑

拒绝。它会让空正文和不断生成的新 UUID 造成无界数据库增长。30 天墓碑窗口覆盖正常迟到请求，配合每用户每小时 60 个新会话上限和 Worker 每批 500 条清理控制增长；`active` 草稿与绑定最终内容的 `published` 会话仍保留，不参与该清理。

### 管理员可以读取所有草稿

拒绝。管理角色用于公开社区治理，不是读取未发布私人内容的授权。需要法务保全或安全调查时必须另建带审计、最小权限和通知规则的 ADR，不能复用普通后台列表。

## 后果与边界

- 正面：同一编辑会话的重试只产生一个 Topic 或一个回复；迟到保存不能覆盖高 revision、复活已清空草稿或越过发布终态；刷新能从 PostgreSQL 找回已提交结果。
- 代价：每次回复草稿写入多维护一条小型会话记录；History 恢复会产生有限的只读查询；跨标签页冲突要求人工刷新，而不是自动合并正文；已清空/接管会话的迟到写入保护窗口明确限定为墓碑最近更新后的 30 天。
- 两个彼此独立、从未共享 key 且都明确提交的编辑会话仍被视为两次用户请求。NextBuf 不按正文相似度合并帖子，也不删除或自动合并升级前已经存在的重复草稿。
- 在线成员、插件、支付、私信和未来功能不在本 ADR 范围内。

## 迁移与回退

`20260720090000_editor_session_idempotency` 是只追加迁移：为 Topic/Post/ReplyDraft 增加 nullable 字段和约束，新增回复会话表及作者/创建时间、状态/更新时间维护索引；既有数据保持 `NULL/NULL`，不回填 key、不修改 UID、Topic number、楼层或历史修订，也不删除草稿。

迁移随 `v0.13.8` 冻结哈希，并必须从精确 `v0.13.7` 镜像执行真实备份、升级和旧内容保留测试。迁移成功后，旧 `v0.13.7` 镜像的迁移全集与数据库不一致，不能作为普通代码回退；失败恢复遵循 ADR-0015，使用升级前 PostgreSQL/附件/配置备份，或发布向前修复。
