# ADR-0003：一个镜像，Web/Worker 分离运行

- 状态：Accepted
- 日期：2026-07-12

## 背景

NextBuf 需要处理邮件、通知、图片、统计、信任计算和周期维护。这些任务不应阻塞 Web 请求。Discourse 常见部署会在一个应用容器内通过进程监督器运行 Web 和 Sidekiq；Rhex 则让同一镜像以不同命令启动 Web 和 Worker。

容器数量不反映系统规模或成熟度。需要选择的是进程监督、故障隔离和扩容边界。

## 决策

- 构建一个包含 Web、Worker 和迁移入口的 NextBuf 镜像。
- 默认 Compose 启动四个常驻容器：Web、Worker、PostgreSQL、Redis。
- Web 和 Worker 使用相同镜像、版本和业务配置，但运行不同命令。
- 数据库 setup/migrate 是执行成功后退出的一次性服务。
- 非 Docker 部署也由 systemd 或 PM2 分别监督 Web 与 Worker。

## 备选方案

### 三容器：Web 与 Worker 同应用容器

容器列表更短，但需要额外引入 runit、Supervisor 或可靠的父进程，正确处理两个子进程的重启、信号、日志和健康状态。Worker 故障和资源峰值也更容易影响 Web。

### 两个独立应用镜像

可以减小各自内容，但会产生两套构建、标签和发布同步问题。Web 与 Worker共享大部分代码，首版收益有限。

### Web 请求内执行后台任务

实现简单，但邮件、图片和批处理会拉长请求，并在超时或进程重启时丢失工作，不可接受。

## 后果

正面影响：

- Worker 可以独立重启、限制资源和水平扩展。
- Web 健康状态不被某个长任务直接绑架。
- 用户仍只拉取一个应用镜像和导入一个 Compose 项目。
- 运维日志和告警能区分请求故障与任务故障。

负面影响：

- 管理面板中会看到四个常驻容器，而不是三个。
- Web 和 Worker 更新时要确保版本一致。
- 本地开发和 CI 必须覆盖两个运行角色。

## 迁移与回退

如果未来提供三容器兼容模式，必须加入可靠进程监督器，并证明信号、重启、健康检查和日志行为正确。该兼容模式不能改变队列协议或数据库数据，因此用户可以随时返回四容器拓扑。
