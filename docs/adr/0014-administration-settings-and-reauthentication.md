# ADR-0014：管理后台、站点设置与管理员二次验证

- 状态：已接受
- 日期：2026-07-16
- 适用版本：`v0.11.0`

## 背景

`v0.10.0` 已具备举报、治理、角色、制裁和信任等级领域服务，但只有聚焦案件与 Worker 的运维页面。日常运营还需要仪表盘、用户和内容工作台、站点设置、Provider 检查以及统一审计查询。后台不能成为绕过领域规则的通用数据库 CRUD，也不能把部署密钥发送到浏览器。

高风险操作仅靠“按钮是否显示”或前端确认弹窗不足以构成安全边界。管理员账号被遗留在已解锁浏览器中时，角色、规则和全站设置仍需额外保护。

## 决策

### 1. 后台仍属于模块化单体

后台页面位于 `src/app/admin`，Route Handler 只解析不可信输入并调用 `server-only` 应用服务。授权、状态转换、事务和审计均在服务端领域边界完成，不新增独立前端工程或通用表编辑接口。

站点管理员可以访问完整后台；节点版主和全局版主只能访问 ADR-0013 已授权范围内的治理工作台。信任等级永远不授予后台权限。

### 2. 在线设置与引导配置分层

PostgreSQL 的单例 `SiteSetting` 保存可在线生效的运营设置：

- 站点名称和注册策略；
- 主题、回复、附件开关；
- 每用户每小时主题、回复和附件上限；
- 乐观并发修订号和最后修改者。

设置采用显式列、数据库 CHECK 和服务端 Zod 校验，不使用任意键值 JSON。写入锁定单例行，校验客户端读取的修订号，并写入不可变治理审计。

数据库、Redis、`AUTH_SECRET`、`MAIL_PAYLOAD_KEY`、SMTP 凭据、对象存储凭据和 OAuth Client Secret 仍是部署引导配置。它们必须在 Web 与 Worker 启动前一致可用，因此 `v0.11.0` 后台只返回脱敏状态并执行固定目标的服务端连接测试，不在线回显或持久化完整密钥。修改这些值后按运维手册重启相关进程。

### 3. 高风险确认与二次验证

管理员通过 Better Auth 的服务端 `verifyPassword` 验证当前账号密码。成功后仅为当前 Better Auth Session 写入一条短时 `AdminReauthentication`：验证时间和过期时间，不保存密码、不签发另一套登录 Cookie，也不把数据库记录当作新身份系统。

高风险服务同时校验：

1. 当前用户仍为 active 站点管理员；
2. 请求 Session 与二次验证记录完全一致且尚未过期；
3. 请求携带与操作约定一致的确认文本；
4. 输入修订号或目标状态仍然有效。

角色授予/撤销、TL4 人工调整、信任规则激活、站点设置、批量会话撤销和审计导出纳入该边界。密码账号可在后台完成二次验证；仅 OAuth 账号在本版显示明确的“不支持密码二次验证”状态，后续如增加 WebAuthn 或 OAuth step-up 必须补充 ADR。

### 4. 审计查询与导出

后台以只读查询统一呈现 identity、community 和 governance 三类不可变审计事件。筛选、游标分页和导出上限由服务端控制；导出需要二次验证和确认文本。任何疑似密钥、密码、Cookie、Token 或连接串字段在返回浏览器或导出前递归脱敏。

### 5. 后台信息架构与工作区

后台采用两级左侧导航，不把所有功能平铺为一列：一级为概览、社区、治理、运维和系统；二级只显示当前一级模块的可用工作区。站点管理员可见全部已实现模块；全局/节点版主只保留授权范围内的治理案件入口。导航隐藏不是授权边界，所有页面和 Route Handler 仍执行服务端权限检查。

仪表盘只显示指标、告警和待办，不能承载创建或编辑表单。运营对象遵循“列表进入、独立工作区修改”的路径：

- 用户：`/admin/users` 列表进入 `/admin/users/[uid]` 详情和受控操作；用户由身份注册流程创建，不提供后台造号入口。
- 内容：主题和回复分别位于 `/admin/content/topics`、`/admin/content/replies`，旧 `/admin/content` 重定向到主题列表；处置继续复用已有主题/治理授权流程。
- 节点：`/admin/nodes` 为列表，`/admin/nodes/new` 为创建，`/admin/nodes/[slug]` 为编辑。已有关联主题或审计的节点不提供硬删除，使用归档保留历史可解释性。
- 系统：站点运营设置、Provider 诊断与信任规则分别位于 `/admin/settings`、`/admin/settings/providers`、`/admin/settings/trust`；Provider 页面只显示脱敏状态并允许固定目标的服务端连接测试。

## 后果

- 运营设置在所有 Web 实例上从 PostgreSQL 读取，Redis 清空不影响正确性。
- Provider 凭据不会因后台功能进入浏览器、日志或通用数据库表。
- 高风险动作和当前登录会话绑定，撤销会话会级联清除二次验证状态。
- 动态 OAuth 注册策略由 Better Auth 用户创建钩子与 NextBuf 注册入口共同约束；Better Auth 继续拥有账号、密码、OAuth、Session 和 Cookie 语义。
- 新增设置必须先判断它属于在线运营设置还是启动引导配置，并同步 Schema、迁移、校验、审计和文档。
- 新后台页面应先归入现有一级模块，并优先补全列表、详情和受控操作路径；不得为了视觉完整性提前显示未实现的产品模块。

## 回退

回退应用前先将在线运营设置恢复为默认值。数据库表可以保留而不影响旧版本；如必须删除，先删除 `admin_reauthentications`，再删除 `site_settings`。不得在回退中修改 Better Auth 的账号、Session 或 Cookie 表语义。
