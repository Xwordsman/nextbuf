# ADR-0009：公开用户标识与头像存储

- 状态：Accepted
- 日期：2026-07-12

## 背景

`v0.5.0` 需要为内容归属、用户主页、提及和客服识别建立稳定公开身份，同时为单机自托管提供不依赖对象存储的头像能力。用户名变更、OAuth 首次注册、既有用户迁移和账号注销都会影响长期链接与冒用风险，因此不能只依赖昵称或临时 URL。

## 决策

### UID

- `users.id` 继续使用 UUID 作为内部主键。
- `users.uid` 使用独立 PostgreSQL 序列，从 `1000` 起分配，唯一、不可变且永不复用。
- 既有用户在迁移时分配 UID；UID 只用于展示、客服和审核，不作为授权凭据。

### Username

- 用户名规范化为小写 ASCII，长度 3 至 24 位，以字母开头，只允许字母、数字和非连续的内部单下划线。
- 服务端维护保留词集合。当前用户名和历史别名共享占用语义；PostgreSQL 约束触发器使用事务级 advisory lock 串行化同名写入，并跨两张表拒绝不同用户冲突。
- 用户名修改后进入 30 天冷却期。旧用户名永久属于原用户，并由 `/u/<alias>` 重定向到当前主页。
- 既有用户迁移为 `user_<UID>`。OAuth 首次创建用户时根据邮箱本地部分生成候选，并加入随机后缀直到满足规则且未占用。
- 只有 `active` 用户能被公开资料页解析；pending 账号不会因用户名可猜而提前公开。

### Profile 与信任展示

- `profiles` 与 `users` 一对一，迁移回填既有用户，并由数据库触发器覆盖 Better Auth、OAuth 和未来受控写入产生的新用户。
- `User.image` 保留为头像 URL，以兼容 Better Auth；简介、主页与隐私偏好进入 `Profile`。
- `v0.5.0` 只把新用户显示为 `TL0`，不持久化或计算信任等级，也不据此执行服务端授权。完整信任系统仍属于 `v0.10.0`。

### 头像存储

- 浏览器执行居中正方形裁剪和缩放，输出 512×512 WebP。
- 服务端限制最终字节数，并独立检查 PNG、JPEG 或 WebP 文件签名；客户端 MIME 和文件扩展名不可信。
- `v0.5.0` 只实现 `local` Provider，使用随机不可变文件名和长期 immutable 缓存。数据库更新失败时清理新文件，更新成功后清理旧的本地头像。
- 本地目录必须持久化。多 Web 实例不能各自使用独立本地目录，进入多实例阶段前必须切换可靠共享文件系统或实现 S3 Provider。

### 注销申请

- 用户提交注销后记录请求时间和 14 天后的计划处理时间，期间可以撤销；重复申请不延长截止时间。
- `v0.5.0` 不立即清除内容、凭证、UID 或历史用户名。最终匿名化、会话撤销、法务保留和备份延迟清理在公开 Beta 前另行实现并记录决策。

## 备选方案

### 只使用 UUID 或昵称

UUID 不适合日常展示和客服沟通；昵称可重复且可修改，不能稳定承载提及和链接。

### 允许释放历史用户名

可以减少名称占用，但会让旧链接和历史提及指向新所有者，产生冒用风险。永久别名更符合社区内容的长期可解释性。

### `v0.5.0` 直接要求 S3

对象存储更适合多实例，但会显著提高自用和宝塔单机安装门槛。先通过 Provider 边界实现本地存储，在真正进入多实例前补充 S3 和迁移工具。

### 立即硬删除账号

实现简单，但会破坏内容归属、审计证据、别名保留和撤销能力，也无法定义备份中的隐私清理。先记录可撤销申请更稳妥。

## 后果

- UID 序列和公开用户名成为长期兼容合同，后续迁移不得重新编号或静默释放别名。
- 用户名修改需要同时维护当前值、别名和冷却时间，并用事务处理并发冲突。
- 本地头像目录成为必须备份和持久化的事实数据；Redis 或构建产物不能替代它。
- 未来删除用户时不能直接依赖当前外键级联，需要先落实历史用户名和内容匿名化政策。
- S3 Provider、头像后台清理任务和最终注销执行器仍是明确后续工作。

## 迁移与回退

迁移 `20260712211500_user_profiles` 为既有用户分配 UID 和 `user_<UID>`，创建 Profile、UsernameAlias、唯一索引和 Profile 自动创建触发器。执行后旧代码不理解必填 UID/username，不能仅切回 `v0.4.0` 镜像；回退需要恢复迁移前 PostgreSQL 备份，并同步恢复头像目录状态。
