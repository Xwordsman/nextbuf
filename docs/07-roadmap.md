# 版本路线图

路线图按能力和兼容承诺组织，不按日期制造不可靠的截止时间。每个版本进入开发前应拆成可验收的里程碑。

具体版本顺序、交付清单和验收门槛见 [详细开发计划](./09-detailed-development-plan.md)。本文只保留高层产品方向；两者发生冲突时，以经过更新的详细开发计划和决策台账为准。

## 1. 基础阶段：工程可运行

目标是建立不会在 V1 中途推倒重来的骨架：

- 初始化 Next.js、TypeScript、Tailwind 和组件系统。
- 建立 PostgreSQL、Prisma、Redis、BullMQ。
- Web、Worker、setup 使用一个镜像和不同入口。
- 完成环境校验、日志、健康检查和请求 ID。
- 建立 CI、迁移测试和完整 Compose 冒烟测试。
- 将 `UI/index.html` 拆为真实页面组件和设计 token。

退出条件：空库可以一键初始化，Web 与 Worker 独立运行，CI 可构建双架构镜像。

## 2. V1.0：可运营的核心社区

### 里程碑 A：身份与基础内容

- 注册、验证、登录、会话和用户资料。
- UID、`@username`、昵称和头像。
- 节点、主题、回复、编辑历史和软删除。
- 首页、节点页、主题页和用户页。

### 里程碑 B：互动与通知

- 收藏、关注、点赞和基础阅读状态。
- 提及、回复、关注事件的站内通知。
- Worker 邮件投递、Outbox、重试和失败查看。
- PostgreSQL 基础搜索。

### 里程碑 C：治理与后台

- 举报、治理案件、处置和制裁。
- 管理员、全局版主、节点版主权限。
- TL0 至 TL4 信任规则、计算历史和重算。
- 用户、节点、内容、举报、设置和审计后台。

### 里程碑 D：发布与加固

- 宝塔 Compose、安装向导和管理员初始化。
- GHCR amd64/arm64 镜像及非 Docker 发布包。
- 备份恢复、升级回滚和安全检查。
- 性能基线、无障碍、移动适配和核心 E2E。

V1.0 的标准不是“页面很多”，而是核心社区可以安全运营、升级和恢复。

## 3. V1.x：稳定和运营能力

在不破坏 V1 API 和部署方式的前提下增加：

- 更完善的反垃圾、敏感操作风控和审核效率。
- 内容摘要、邮件摘要、订阅和运营统计。
- 节点规则、主题模板和细粒度通知偏好。
- S3、SMTP、OAuth Provider 的后台诊断。
- 导入导出、数据保留和隐私工具。
- PostgreSQL 搜索质量和性能优化。

## 4. V2：开放能力与服务场景

V2 在社区稳定后增加：

### 开放 API

- 个人和应用 Token、Scope、撤销与审计。
- 稳定的 `/api/v1` 合同、限流和 Webhook。
- OpenAPI 文档和官方 SDK 的生成策略。

### 专业声誉

- 与 TL 分离的领域身份、作品或资质展示。
- 可验证的贡献记录和反作弊机制。
- 不以付费或管理角色直接换取专业评分。

### 服务信息与交易前置基础

- 服务发布、咨询意向、评价资格和治理规则。
- 交易信用与社区信任分离。
- 为后续订单、账务和支付保留稳定主体标识与幂等接口。
- V2 不处理真实资金；在明确法务、退款、风控和地区限制前，不上线支付或资金托管。

### 搜索和运营

- 可选独立搜索 Provider。
- 更完善的内容推荐、趋势和反操纵规则。
- 运营自动化与细粒度权限。

## 5. V3：插件平台

### 插件平台

- 插件 Manifest、兼容版本和能力声明。
- 安装、启用、停用、升级、回滚和卸载生命周期。
- 受控的 UI 插槽、领域事件、Provider 和后台页面扩展。
- 插件数据库迁移、命名空间、任务和配置管理。
- 签名、来源、权限审查和故障隔离策略。
- 插件市场只分发经过元数据和兼容检查的包。

V3 之前不承诺插件 API 稳定。V1/V2 的内部模块接口可以为插件做准备，但不能提前暴露全部内部对象。

## 6. V4 及以后：平台深化

- V4：订单、支付、退款、争议和交易信用。
- V5：主题、扩展市场和生态工具。
- V6：多 Web、多 Worker、高可用、可观测性和灾难恢复。
- V7：组织空间和多社区能力。
- V8：数据可携带和受控联邦研究。
- V9：可控、透明、尊重隐私的 AI 辅助能力。
- V10：LTS、迁移体系、兼容承诺和生态治理。

### 多节点/联邦研究

“多节点”可能指多应用实例、地域部署或社区联邦，三者架构完全不同。进入对应大版本设计前必须先定义需求，不在 V1 数据模型中预埋未经验证的分布式复杂度。

## 7. 兼容与演进原则

- V1 内部模块可调整，但公开 API 从发布后遵守弃用周期。
- 数据迁移优先向前兼容，跨多个小版本升级路径必须测试。
- 插件兼容以明确 API 版本为准，不承诺访问内部 Prisma Model。
- Compose 服务名、卷名和必需环境变量变更需要迁移说明。
- 安全修复可以优先于兼容性，但必须清楚记录影响和恢复步骤。

## 8. 每个大版本开始前必须回答

1. 用户问题是什么，成功如何衡量？
2. 哪些数据一旦产生就必须长期兼容？
3. 哪些权限、安全和法务风险新增？
4. Worker、队列、存储和备份如何变化？
5. 从上一个稳定版如何升级和回滚？
6. 哪些能力属于核心，哪些应成为 Provider 或插件？
