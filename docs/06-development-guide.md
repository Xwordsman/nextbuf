# 开发指南

第一次搭建环境、启动依赖和执行命令见 [本地开发手册](./11-local-development.md)。本文件重点规定代码、数据库、API、队列和测试的工程约束。

## 1. 仓库结构

初始化工程时建议采用以下结构：

```text
NextBuf/
  src/
    app/
      (site)/
      admin/
      api/
    components/
      ui/
      community/
      admin/
    modules/
      identity/
      profiles/
      community/
      interactions/
      notifications/
      moderation/
      trust/
      settings/
      audit/
    infrastructure/
      database/
      queue/
      cache/
      mail/
      storage/
      search/
    worker/
    shared/
  prisma/
  public/
  tests/
  deploy/
  docs/
  UI/
```

这是已确定的顶层结构，不创建顶层 `frontend/` 和 `backend/`。Next.js 页面、Server Components、Route Handlers、领域服务和 Worker 属于同一个应用，只按职责分层。完整目录责任和导入规则见 [仓库结构与模块边界](./10-repository-structure.md)。

不要求为每个小功能创建一套空目录。模块内部按用例、领域和数据访问需要组织，避免整个项目只有全局 `services/`、`utils/` 和 `types/`。

## 2. 依赖方向

- 页面和接口依赖应用用例。
- 应用用例依赖领域规则和仓储接口。
- Prisma、Redis、邮件和对象存储属于基础设施实现。
- 领域模块不直接读取 `process.env`。
- Worker 调用与 Web 相同的应用服务，不复制业务逻辑。
- 模块不能跨边界直接修改对方的数据表；通过公开服务或领域事件协作。
- `src/modules`、`src/infrastructure` 和 `src/worker` 默认为服务端代码，入口必须使用 `server-only` 保护。
- Client Component 不得导入 Prisma、Redis、队列、环境密钥或服务端模块。

禁止循环依赖。需要共享的稳定值进入 `shared`；只被两个模块偶然共用的逻辑不应立即抽成全局工具。

## 3. TypeScript 和输入边界

- 启用严格模式，不使用无说明的 `any`。
- 外部输入一律视为 `unknown`，在边界通过 Schema 解析。
- 数据库实体、API DTO 和页面 ViewModel 不使用同一个万能类型。
- 金额、时间、ID 和枚举使用明确类型，禁止用展示字符串参与业务判断。
- 环境变量在进程启动时集中解析，代码中不散落非空断言。

## 4. Next.js 约束

- 默认使用 Server Components；只有真实交互需要时才标记 Client Component。
- 缓存和 revalidation 必须显式设计，涉及用户权限的响应不得误用公共缓存。
- Server Action 适合站内表单，不作为公开 API 合同。
- Route Handler 使用统一错误、认证、限流和请求 ID 中间层。
- 管理后台写操作仍通过应用服务授权，不能直接在组件中调用 Prisma。
- Node-only 的数据库、队列和原生依赖不能被打包到 Edge Runtime。
- 文件需要明确强调运行边界时使用 `*.server.ts` 或 `*.client.tsx`；不能只依赖开发者记忆判断导入是否安全。

## 5. 数据库开发

### Schema

- 表名、字段名和索引遵循统一命名规则。
- 必需关系使用外键；唯一业务规则尽量使用数据库约束。
- 列表页查询所需索引与功能一起提交。
- 软删除唯一性、部分索引和全文搜索可使用 PostgreSQL 原生迁移 SQL。

### 迁移

- 每次迁移进入版本控制并在空库、旧版本样本库上测试。
- 生产启动不由每个 Web 副本争抢执行迁移；由一次性 `setup/migrate` 任务执行。
- 大表变更评估锁表、回填和索引构建时间。
- 数据回填使用可恢复、可观察的批任务，不把长时间脚本塞进单个事务。
- 禁止修改已经发布的迁移来伪造历史，应创建修复迁移。

### 查询

- 默认通过仓储或模块数据访问层使用 Prisma。
- 原生 SQL 必须参数化、具备测试并说明 Prisma 无法合理表达的原因。
- 列表查询避免 N+1；分页默认使用稳定游标和确定性排序。

## 6. API 设计

- V1 内部接口与公开 API 分开命名和鉴权。
- 公开 API 从 `/api/v1` 开始版本化。
- JSON 字段和错误码一旦公开，按兼容政策维护。
- 列表提供明确分页上限，不能允许无界导出。
- 幂等创建操作支持 Idempotency Key，特别是未来交易与 Webhook。
- 错误响应包含稳定代码、可读信息和请求 ID，不暴露内部堆栈。

## 7. 队列任务规范

每个任务定义：

- 稳定名称和 payload 版本。
- 幂等键和最大重试次数。
- 指数退避、超时和失败分类。
- 是否允许并发、顺序要求和资源上限。
- 失败后的人工处理或死信策略。

任务 payload 不保存密码、访问令牌或大段用户内容。Worker 必须假设同一个任务可能执行多次。

周期任务使用分布式锁或 BullMQ 调度能力，确保多 Worker 时不会重复产生不可幂等副作用。

## 8. 测试策略

### 单元测试

覆盖纯领域规则：权限、信任计算、状态机、热门度、限流策略和内容规则。

### 集成测试

使用真实 PostgreSQL 和 Redis，覆盖：

- 唯一约束、事务和迁移。
- Outbox 到队列再到 Worker 的完整链路。
- 会话撤销、权限和制裁。
- 搜索、分页和软删除。

### 端到端测试

Playwright 覆盖核心用户旅程：

- 注册/登录、发主题、回复和编辑。
- 通知、举报和版主处置。
- 管理员设置与权限拒绝。
- 桌面和移动端关键页面。

### 部署测试

CI 使用发布镜像启动完整 Compose，验证 setup、Web、Worker、PostgreSQL 和 Redis。至少验证一次优雅停止与任务恢复。

测试数量不代替风险覆盖。权限和迁移变更必须增加对应回归测试。

## 9. UI 与可访问性

- 组件优先复用 shadcn/ui 原语和现有设计 token。
- 列表、工具栏和表单在常见移动/桌面宽度下不得重叠。
- 不依赖颜色单独表达置顶、错误或信任状态。
- 交互元素支持键盘、可见焦点和正确语义。
- 图标按钮提供可访问名称和必要提示。
- 动画尊重 `prefers-reduced-motion`。
- 静态稿改动同步记录到产品或设计文档，避免 UI 与数据能力脱节。

## 10. 日志与错误

- 使用结构化日志，不在业务代码随意 `console.log`。
- 预期业务失败使用稳定错误类型；未知异常进入统一错误边界。
- 日志包含 correlation/request ID，异步任务继续传递该关联 ID。
- 不记录 Cookie、Authorization、密码、完整邮箱验证码或 Provider 密钥。
- 用户可见错误用明确语言说明可采取的动作，内部细节只进入安全日志。

## 11. Git 与合并要求

- 主分支始终保持可构建和可迁移。
- 每个变更范围尽量单一，生成文件和源文件一起提交。
- 合并前通过格式、lint、类型检查、单元测试和相关集成测试。
- 数据库、API、权限、部署变更必须更新对应文档。
- 新依赖需要说明用途、维护状态、许可证、包大小或运行风险。
- 不提交 `.env`、数据库转储、真实上传和临时调研仓库。

## 12. Definition of Done

一个功能只有同时满足以下条件才算完成：

- 行为和验收标准明确。
- 服务端授权与输入校验完成。
- 数据迁移可从受支持旧版本执行。
- 正常、拒绝和失败路径有相应测试。
- 日志、审计和任务失败可以定位。
- 桌面与移动界面通过检查。
- 配置、API、部署或用户行为变化已更新文档。
- 不引入已知的高危依赖或明文秘密。
