# 公开 Beta 人工验收模板

本文用于 `v0.13.x` 邀请测试和生产候选验收。每次安装、升级或恢复演练复制一份记录到发布工单或私有运维系统；不要把密码、Cookie、Token、完整邮箱、连接串或备份密钥写入公开 Issue。

## 1. 验收信息

| 字段 | 记录 |
| --- | --- |
| 验收人 / 日期 |  |
| 应用版本 / Git SHA |  |
| 镜像精确标签 / Digest |  |
| 部署方式 | 宝塔单文件 Compose / CLI 受控 Compose / 非 Docker |
| 架构 | amd64 / arm64 |

| 开始 / 完成时间 |  |
| 关联工单 / 证据位置 |  |

## 2. 记录格式

每个步骤使用同一行格式：

| ID | 操作 | 预期 | 实际 / 证据 | 结果 | 问题 |
| --- | --- | --- | --- | --- | --- |
| 示例 | 访问 `/health/ready` | HTTP 200，数据库和 Redis 就绪 | 截图或日志位置 | 通过 / 失败 / 阻塞 | Issue 链接 |

问题级别只能使用：`阻断`（安全、数据损坏、无法安装/升级/恢复或核心旅程不可用）、`重要`（明显影响运营但有可靠绕行）、`增强`（不阻止 Beta）。

## 3. 首次安装

- [ ] 宝塔使用发布包中的 `compose.baota.yml` 且不依赖 `.env`；CLI 受控部署使用 `compose.yml`、`.env.example` 和精确版本。
- [ ] 宝塔模板中的 `latest` 来自通过完整检查和双架构基础烟测的 `main`；记录镜像 Digest、`/api/version` 的 Git commit 和源码 SemVer，不能只记录可变标签。
- [ ] PostgreSQL、Redis 不发布公网端口；Web 只绑定回环地址并由 HTTPS 反向代理。
- [ ] `nextbufctl init` 或等价面板配置生成独立强密钥，示例值全部替换。
- [ ] 空数据库通过默认 Compose 自动完成 setup；Web 未健康前 Worker 不启动，编排中只有四个常驻容器且没有停止的 setup 记录。
- [ ] 首次访问根域名以 307 自动进入 `/setup`；安装完成后首页正常打开。
- [ ] `/setup` 创建唯一首位管理员；重复请求被拒绝，空安装没有预置业务节点。
- [ ] 首位管理员能在 `/admin/nodes` 创建当前社区自己的第一个节点，重复 slug 和非管理员创建均被拒绝。
- [ ] 删除 `SETUP_TOKEN` 并重启 Web，`doctor` 全部通过。
- [ ] `Powered by NextBuf` 在桌面和移动端可见且链接正确。

## 4. 身份与核心社区旅程

- [ ] 邀请或开放注册、邮箱验证、登录、退出、找回密码和旧会话撤销。
- [ ] 修改昵称、`@username`、简介、隐私和头像；旧用户名正确重定向。
- [ ] 创建主题、Markdown 预览、上传图片/文本/PDF/ZIP，并验证刷新后附件仍在。
- [ ] 输入正文后在自动保存开始前立即发布；只产生一个已发布 Topic/回复，账号草稿列表不残留同一内容。
- [ ] 在自动保存显示“正在保存”时继续修改并立即发布；最终页面必须是最后输入的正文，刷新后不回退到旧快照。
- [ ] 上传附件期间继续输入并切换预览；发布按钮保持禁用，上传完成后文字与附件 Markdown 都保留且可发布。
- [ ] 在代理或浏览器中让一次发布已经提交但返回无效 200 JSON，再刷新/返回；界面先提示结果未知，随后恢复到同一 Topic 或楼层，不创建重复内容。
- [ ] 清空回复草稿并等待保存，再刷新和继续回复；旧正文不能重新出现。新会话接管后返回旧 history 项应清除旧会话并回到规范 Topic 页面状态，不能重复刷新；两个仍有效的标签页冲突时，较旧页面应提示冲突并要求刷新，不能静默覆盖。
- [ ] 先保存回复草稿，再分别关闭 Topic 和限制作者账号；两种状态下作者仍能删除自己的既有草稿，删除能力不随发布权限一起丢失。
- [ ] 设置 `repliesEnabled=false` 后仍可保存和编辑回复草稿，但正式发布必须被拒绝；恢复开关后可发布最后保存的草稿。
- [ ] 回复、引用、提及、编辑、软删除和恢复；楼层不重排。
- [ ] 点赞、收藏、关注、阅读状态、搜索和通知未读/已读旅程。
- [ ] 390、1024、1440 宽度无水平溢出；键盘可跳到主要内容并完成关键表单。

## 5. 治理与后台

- [ ] 普通用户能举报主题、回复和用户，不能访问管理 API。
- [ ] 节点版主只能处理所属节点；全局版主和管理员作用域正确。
- [ ] 制裁在服务端阻止对应发布行为，撤销后恢复。
- [ ] 高风险管理员操作要求当前 Session 二次验证和确认文本。
- [ ] 站点设置、Provider 诊断、审计查询/导出和 Worker 页面可用且不泄漏密钥。
- [ ] 创建活动主题草稿、删除自草稿和一条隔离构造的删除来源未知记录；管理员/版主的主题/回复列表、搜索、仪表盘、用户/节点计数、详情和审计导出均不能看到它们，按数字编号尝试修改/删除/恢复/管理均与不存在主题一样返回 404，作者的 `/account/topics` 仍可管理。
- [ ] 停止 Worker 或制造队列积压后后台显示真实告警，不伪造在线状态。

## 6. 升级

- [ ] 记录旧版本、目标版本、当前迁移和镜像 Digest，确认发布说明允许直接升级。
- [ ] 升级前 `doctor` 通过，备份归档校验成功并复制到异机。
- [ ] `nextbufctl upgrade <精确版本>` 自动拉取、停写、运行目标 setup 并恢复 Web/Worker。
- [ ] 升级后版本接口、管理员、用户会话、主题、回复、附件、通知和后台仍可用。
- [ ] `v0.13.7 -> v0.13.8` 后抽查历史 Topic/首帖/回复/ReplyDraft 的编辑 key/revision 为 `NULL/NULL`，公开编号和楼层不变，附件当前/修订/草稿引用仍在；新发布内容可恢复编辑会话且不会留下重复草稿。
- [ ] 数据库迁移无失败/未知项，`runtime.initialized` 为目标版本，Outbox/Queue 持续下降。

## 7. 空卷恢复

- [ ] 在隔离环境验证归档 SHA-256，使用空 PostgreSQL/Redis/附件卷执行恢复。
- [ ] 比较恢复前后的用户、主题、回复、附件记录数量与抽样内容。
- [ ] 本地附件可读取；S3 模式已单独验证 Provider 快照/版本恢复。
- [ ] Redis 不从备份恢复，Worker 能由 PostgreSQL Outbox/调度事实重建执行状态。
- [ ] 恢复后 setup、Web、Worker、`doctor` 和首次管理员完成状态正常。

## 8. 验收结论

| 项目 | 记录 |
| --- | --- |
| 阻断问题 |  |
| 重要问题及绕行 |  |
| 增强建议 |  |
| 已知限制已确认 | 是 / 否 |
| 安装可接受 | 是 / 否 |
| 升级可接受 | 是 / 否 / 不适用 |
| 恢复可接受 | 是 / 否 |
| 最终结论 | 通过 / 有条件通过 / 不通过 |

任何未关闭的阻断问题都使本次验收不通过。自动 CI、axe、容器冒烟和 `doctor` 是必要证据，但不能替代真实域名、SMTP、对象存储、备份位置和运营角色下的人工验收。
