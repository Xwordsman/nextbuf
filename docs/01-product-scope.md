# 产品定位与范围

## 1. 产品愿景

NextBuf 是一个围绕 AI、建站、主机、域名和互联网技术服务建立的开放社区。它首先是一套可长期运营的社区程序，其次才是内容展示网站。

产品需要兼顾：

- 新用户快速找到可靠信息。
- 活跃成员持续交流、沉淀经验并获得合理权限。
- 站长和版主高效治理垃圾信息、争议和滥用。
- 开源部署者以较低成本完成安装、升级、备份和二次开发。
- 后续业务可以加入服务交易、开放 API 和插件，而不破坏基础社区模型。

## 2. 目标用户

### 普通访客

- 浏览节点、主题和用户公开资料。
- 使用搜索和排序寻找解决方案。
- 了解社区规则后注册。

### 注册成员

- 发布主题和回复。
- 收藏、关注、点赞或表达其他受控反馈。
- 接收站内与邮件通知。
- 随参与质量提升信任等级。

### 专业参与者

- 分享 AI 工具、建站经验、主机测评和域名知识。
- 在后续版本建立可验证的专业声誉。
- 在交易功能上线后提供或购买服务。

### 版主与管理员

- 管理节点、内容、用户和举报。
- 查看审计记录和风险信息。
- 配置站点、登录方式、邮件、存储和信任规则。

### 开源部署者与贡献者

- 通过宝塔 Docker Compose 或命令行安装。
- 使用外部数据库、Redis、对象存储和邮件服务。
- 开发主题、集成或后续版本的插件。

## 3. 产品原则

1. **阅读优先**：首页保持高信息密度，标题、节点、作者、时间和互动数据易于扫描。
2. **渐进增强**：V1 先把讨论、治理、通知和部署做稳定，再增加交易与插件。
3. **身份信息清晰**：昵称、`@username`、不可变 UID 和信任等级各自承担不同用途。
4. **治理可解释**：删除、封禁、降级等管理行为必须记录原因和操作者。
5. **默认安全**：敏感能力需要显式授权；插件和外部回调不能绕开权限层。
6. **可自托管**：核心社区能力不依赖某个专有云才能运行。

## 4. V1 必须完成的能力

### 账号与身份

- 注册、登录、退出、找回密码和邮箱验证。
- 唯一用户名 `@username`、可修改昵称、不可变数字 UID。
- 头像、简介、基础隐私设置和会话管理。
- 至少支持密码登录；OAuth 提供者按配置启用。

### 社区内容

- 节点的创建、排序、显示和权限控制。
- 主题发布、编辑、预览、关闭、置顶、精华和热门状态。
- 回复、楼层定位、引用、编辑历史和软删除。
- 首页、节点页、主题页、用户页和基础搜索。
- 收藏、关注、点赞及基础浏览记录。

### 通知与异步任务

- 回复、提及、关注主题变化和管理动作的站内通知。
- 验证、重置密码和重要通知邮件。
- 邮件、内容处理、摘要等耗时任务通过 Worker 执行。
- 任务必须可重试并防止重复副作用。

### 治理

- 举报主题、回复和用户。
- 管理员、版主和普通用户的角色权限。
- 用户警告、限流、禁言、封禁和申诉备注。
- 管理操作审计日志。
- 可配置的信任等级计算和能力门槛。

### 管理后台

- 仪表盘、用户、内容、节点、举报、日志和站点设置。
- 后台与前台同一应用、同一认证体系和同一部署产物。
- 高风险操作要求再次确认并写入审计日志。

### 安装与维护

- amd64 和 arm64 容器镜像。
- 宝塔可导入的 Docker Compose 文件。
- 数据库迁移、初始化管理员、健康检查、备份和升级说明。
- 非 Docker 发布包和 systemd/PM2 运行示例。

## 5. V1 明确不做

- 完整交易市场、担保、支付和结算。
- 任意第三方服务端插件的在线安装与执行。
- 用户私信和群聊。
- 用户自定义标签系统。
- 多种表情反应；V1 只提供单一点赞。
- 多租户 SaaS。
- 跨站联邦协议。
- 独立搜索集群作为强制依赖。
- 原生移动 App。
- 为兼容多种数据库而牺牲 PostgreSQL 能力。

这些内容不是永久排除，而是不能阻塞 V1 交付。

## 6. 首页静态稿约束

当前 `UI/index.html` 确定了以下方向：

- 采用三栏桌面布局，保持既定总宽度和左右栏尺寸。
- 首页与公开主题正文页使用同一三栏社区壳层；主题页只将中栏替换为正文、回复和回复编辑器，左侧节点导航与右侧社区面板保持可用。
- 主题页首帖使用单一连续内容模块，依次呈现站点名称与网址、节点、标题、发帖人链接与发布时间及浏览数、正文和主题操作，不把题头与正文拆成两张卡片。
- 每条回复左侧仅呈现链接到成员页的头像；右侧首行依次呈现回复者名称链接和回复时间，并将稳定楼层永久链接放在最右侧，正文及回复操作位于其下。
- 风格接近 V2EX 的信息密度以及 shadcn/ui、Vercel 的克制视觉。
- 帖子标题右侧显示置顶、精华、热门等状态。
- 第二行以灰色标识显示节点，再显示昵称、发布时间和浏览量；浏览量后紧跟最后回复人及其时间。
- 帖子列表昵称旁不显示信任等级。
- 头像菜单显示昵称、`@username`、UID，信任等级位于 UID 右侧。
- 所有标准 Web 页面底部保留清晰可见的 `Powered by NextBuf` 官方项目链接；部署者可以在旁边添加自己的社区版权信息。

这些是信息架构和表现规则，不意味着后端直接保存“热门”等所有展示结果；部分状态可由规则计算。

## 7. 成功标准

V1 发布前至少满足：

- 新部署者按照文档可完成安装、初始化、升级和恢复演练。
- 核心发帖、回复、通知、举报和管理流程有自动化测试。
- Web 或 Worker 单独重启不会造成已提交内容丢失。
- 权限检查位于服务端，前端隐藏按钮不是唯一保护。
- 常见页面在桌面和移动端无布局重叠，键盘操作和基本无障碍可用。
- 管理员能够定位失败任务、异常登录和关键管理操作。
