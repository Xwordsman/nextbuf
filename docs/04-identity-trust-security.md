# 身份、信任与安全

## 1. 四套概念必须分离

NextBuf 不使用一个等级解决所有治理问题：

| 概念 | 回答的问题 | 示例 |
| --- | --- | --- |
| 身份 | 这个账号是谁 | UID、`@username`、OAuth 绑定 |
| 管理角色 | 站点授予了什么管理职责 | 管理员、全局版主、节点版主 |
| 信任等级 | 正常社区参与解锁了什么能力 | TL0 至 TL4 |
| 专业/交易信用 | 在特定领域是否可靠 | 后续的专业认证、交易评价 |

TL3 用户不自动成为版主；管理员也不应通过伪造阅读天数获得管理权限。首页和用户菜单只展示必要信息，不增加“建站”等所谓专业标签。

## 2. 登录与会话

### 认证实现

认证采用 Better Auth `1.6.23` 与官方 Prisma adapter，决策见 [ADR-0008](./adr/0008-better-auth.md)。Web、Route Handler 和服务端页面共享同一认证实例；不自行实现密码协议、OAuth 状态校验、会话签名或 Cookie 格式。

`v0.4.0` 已实现：

- 邮箱和密码注册登录。
- 邮箱验证、找回密码、单个/其他/全部会话撤销。
- 开放、邀请码、关闭三种注册策略；邀请码只保存 HMAC。
- 可选 GitHub OAuth；未配置时不显示，邀请制/关闭注册时不允许 OAuth 创建新账号。
- 当前用户可以查看自己的登录设备；管理员代管会话属于后续后台版本。
- 关键注册、验证、重置和会话事件写入身份审计表，IP 只保存 HMAC。

`v0.5.0` 在 Better Auth 用户记录上增加 UID、用户名和资料字段，但不改变 Better Auth 的密码、账号、会话、验证或 Cookie 字段语义。邮箱注册由 NextBuf 入口接收用户名；OAuth 首次创建用户时由服务端生成符合相同规则且未被占用的用户名。

公开邮箱注册只能经过 `/api/identity/register`。Better Auth 底层邮箱注册入口要求内部 HMAC 头，防止绕过邀请码和关闭注册策略。有效请求无论邮箱是否已存在都返回同类接受响应，避免账号枚举。

### 密码和令牌

- 密码使用 Better Auth 推荐的 scrypt；不引入原生 Argon2 模块，降低 ARM64、standalone 和容器构建风险。
- 验证与重置标识在 Prisma adapter 外层使用 `AUTH_SECRET` 做 HMAC-SHA256，数据库只保存 64 位十六进制标识；令牌值仍由 Better Auth 管理。
- 登录失败响应不泄露邮箱或用户名是否存在。
- 密码重置撤销全部旧会话；邮箱修改和 OAuth 解绑策略在实现对应功能时补充。

### Cookie

- `APP_URL` 为 HTTPS 时，Better Auth 使用 `Secure`、`HttpOnly` 和合适的 `SameSite`；公网生产环境必须使用 HTTPS。仅 standalone 本机测试允许 loopback HTTP，并相应关闭 `Secure` 以完成真实浏览器认证流程。
- 会话有绝对有效期和更新周期，默认分别为 30 天和 1 天。
- 反向代理后的 HTTPS 判断必须配置可信代理，不能信任任意转发头。

### 身份邮件

- Web 请求只创建 `email_deliveries` 和 Outbox 事件，不同步连接 SMTP。
- 邮件正文以独立 `MAIL_PAYLOAD_KEY` 使用 AES-256-GCM 加密后落库。
- Worker 消费 `nextbuf.identity.email.send@1`，通过 SMTP 发送，并使用稳定 Message-ID 降低重试重复邮件影响。
- `v0.9.0` 的普通通知邮件复用同一加密投递边界，但读取 NotificationPreference；邮箱验证和密码重置是安全邮件，不能被普通通知偏好关闭。
- `AUTH_SECRET` 与 `MAIL_PAYLOAD_KEY` 不能复用；前者轮换会使未使用验证/重置链接失效，后者丢失会使待发送邮件不可解密。

## 3. 用户标识规则

### UID

- 使用独立 PostgreSQL 序列从 `1000` 起分配；既有用户在迁移时按序获得 UID。
- 注册时分配，不可修改、不复用，删除账号也不回收序列值。
- 用于客服、审核和界面准确识别。
- UID 可公开不意味着可作为数据库主键或授权凭据。

### Username

- 输入先去除首尾空白并转为小写，最终长度为 3 至 24 个 ASCII 字符。
- 必须以小写字母开头，只允许小写字母、数字和内部单下划线；不能连续使用下划线，也不能以下划线结尾。
- `admin`、`api`、`auth`、`nextbuf`、`system`、`support`、`user` 等路由、系统和冒充风险名称进入服务端保留词集合。
- 当前用户名与历史别名共同占用命名空间；用户名用于 `@username` 和 `/u/<username>`。
- 修改成功后进入 30 天冷却期。旧名称永久归原用户所有并重定向到当前主页，不设置自动释放期。
- 既有用户迁移为 `user_<UID>`；OAuth 新用户使用邮箱本地部分清洗后加随机后缀，最终仍经过同一校验和占用检查。
- 数据库约束触发器以事务级 advisory lock 串行化同名写入，并跨当前用户名与历史别名拒绝冲突；客户端 `pattern` 只用于即时提示，不能代替服务端和数据库验证。

### Display name

- 昵称可以重复并允许更宽松的字符集。
- 昵称不能代替用户名参与唯一识别。
- 仿冒管理员、控制字符和误导性空白需要限制。

### 资料、头像和注销

- 公开资料允许 500 字简介和 HTTP/HTTPS 个人主页；邮箱始终不出现在公开用户页。
- 未激活账号不提供公开用户页。用户可关闭资料或活动统计展示，但 UID、昵称、用户名和当前 `TL0` 身份仍作为内容归属的基本信息。
- 浏览器把头像居中裁剪为 512×512 WebP 并提供 1 至 3 倍缩放；服务端独立限制字节数并检查 PNG、JPEG、WebP 文件签名，不能信任扩展名或 MIME 声明。
- `v0.5.0` 使用不可变随机文件名和本地存储 Provider。替换头像先更新数据库引用，再清理旧文件；媒体响应使用 immutable 缓存和 `nosniff`。
- 注销是 14 天可撤销申请，重复申请不会延后计划时间。`v0.5.0` 只记录申请，不立即删除内容、凭证或历史用户名；最终处理策略必须在公开 Beta 前结合隐私政策实现。

## 4. 授权模型

权限采用 RBAC 与资源条件结合：

- RBAC：用户、节点版主、全局版主、管理员等角色授予基础能力。
- 资源条件：是否作者、是否属于管理节点、主题是否关闭、制裁是否生效。
- 信任能力：发链接数量、发帖频率、举报权重等社区能力。

统一授权调用形式应接近：

```ts
authorize(actor, "topic.close", topic)
```

禁止仅依赖路由前缀、客户端按钮隐藏或中间件完成最终授权。领域写操作必须在服务端再次授权。

`v0.6.0` 已建立社区授权骨架：

- `community_role_assignments` 支持 `admin`、`global_moderator` 和限定节点的 `node_moderator`，使用 `scope_key` 唯一约束和数据库 CHECK 保证作用域一致。
- 活跃作者可以创建、编辑、软删除和恢复自己的主题；节点版主只管理所属节点的非草稿主题，全局版主和管理员管理全站非草稿主题。
- 私有草稿只对作者可见，版主身份不能自动读取或发布他人草稿。
- 置顶、精华、关闭和隐藏只能由具有管理能力的服务端用例修改；节点可见性、排序和归档只允许管理员修改。
- 所有写入 Route Handler 都执行会话、同源、输入和领域授权检查；数字主题编号只是公开定位符，不是授权凭据。
- 发布事务锁定用户行并查询 PostgreSQL 事实：每小时最多发布 3 个主题。Redis 清空不会放宽该限制；草稿每人最多 20 个。

`v0.7.0` 将同一授权边界扩展到回复和附件：active 用户才能保存回复草稿、回复或上传；关闭主题只允许具有节点管理能力的角色继续回复；作者只能编辑/删除/恢复自己的回复，版主按既有节点作用域管理。回复限流为每用户每小时 20 条，附件限流为每用户每小时 20 个，均通过 PostgreSQL 用户行锁防止并发绕过。

`v0.10.0` 将有效制裁加入同一服务端授权：`site_mute` 关闭全站主题、回复、草稿和附件上传，匹配节点的 `node_mute` 关闭该节点主题、回复和草稿，不误伤其他节点内容或阅读、收藏、关注、点赞；`suspend`/`ban` 立即使账号能力失效并同步 User.status。节点版主只管理所属节点，全局版主不能永久封禁，角色/TL4/规则激活只允许管理员，且撤销最后一个管理员会被事务拒绝。页面隐藏按钮、Redis 缓存和 TL 都不是最终授权来源。

## 5. 信任等级

`v0.10.0` 已持久化并计算信任等级。新账号由数据库触发器创建 TL0 状态；页头、公开资料和 `/account/trust` 读取真实 `TrustUserState`，并展示规则版本、指标、宽限期和历史。TL 仍不授予任何版主、管理员、敏感数据或封禁能力。

### 等级语义

| 等级 | 名称建议 | 典型状态 | 能力方向 |
| --- | --- | --- | --- |
| TL0 | 新用户 | 刚注册或尚未验证参与质量 | 严格频率、链接和提及限制 |
| TL1 | 基础用户 | 完成验证并有基本阅读参与 | 正常回复、基础互动 |
| TL2 | 成员 | 持续正常参与，无近期严重违规 | 更高限额、更多社区功能 |
| TL3 | 资深成员 | 长期活跃且行为稳定 | 更高举报权重、有限协作能力 |
| TL4 | 社区领袖 | 极高信任，通常需要人工确认 | 明确授权的社区协助能力 |

当前版本只使用 TL 作为可解释的社区信任与举报权重信号，尚未按等级提高发帖/链接限额。任何涉及删除、封禁、查看敏感信息的能力始终由独立管理角色控制。

### 计算原则

- 晋级考虑账号时长、有效阅读、正常发帖、获得反馈和违规记录。
- 防止单纯刷访问次数、互赞或短期灌水快速晋级。
- TL0 至 TL3 自动计算；TL4 只能由管理员人工授予和撤销。
- 降级使用时间窗口和宽限期，避免一次正常波动导致频繁升降。
- 严重制裁可以临时冻结部分能力，但不篡改历史等级记录。
- 每次等级变化保存原因、规则版本、前后值和计算时间。

### 规则配置

阈值可配置，但配置必须：

- 有类型、最小值、最大值和默认值。
- 变更前提供影响预估，避免一次设置让大量用户突然升降级。
- 记录操作者并触发分批重算任务。
- 允许规则升级后重放计算，不依赖不可恢复的 Redis 计数。

实际默认规则 v1：

| 等级 | 账号天数 | 已读主题 | 有效发帖 | 收到点赞 | 180 天有效违规上限 |
| --- | ---: | ---: | ---: | ---: | ---: |
| TL1 | 1 | 3 | 1 | 0 | 0 |
| TL2 | 14 | 20 | 10 | 3 | 0 |
| TL3 | 60 | 100 | 50 | 20 | 0 |

晋级立即应用；候选等级低于现有自动等级时先进入 14 天宽限，宽限截止后仍不达标才降级。规则状态依次为 draft、previewed、active/retired；preview 和 apply 均由保存进度与影响分布的 UID 游标批次执行。完整决策见 [ADR-0013](./adr/0013-governance-roles-trust.md)。

## 6. 风控与限流

基础限流维度：IP、账号、会话、动作类型和资源目标。至少覆盖：

- 注册、登录、验证码和密码重置。
- 发主题、回复、提及和上传；未来版本若实现私信，必须增加独立限流与骚扰防护。
- 举报、点赞、收藏和搜索。
- API Token 和 Webhook。

Redis 用于快速计数，关键封禁与制裁保存在 PostgreSQL。代理和 CDN IP 头只接受明确配置的可信来源。

`v0.8.0` 的互动写入要求 active 登录用户和同源请求，复合主键负责点赞、收藏和关注最终唯一性。匿名浏览只保存登录用户 ID 或 IP/用户代理组合的领域分离 HMAC，不保存原始识别信息；同一 Topic/访问者/30 分钟桶只接受一次。搜索 SQL 全部参数化，并在查询层过滤隐藏、删除和非公开内容。详细边界见 [ADR-0011](./adr/0011-interactions-search-discovery.md)。

`v0.10.0` 的举报写入同样要求 active 登录用户和同源请求，并在 PostgreSQL 锁定举报者：同一举报者/目标只有一条未结举报，每用户 UTC 日最多 10 条。相同目标的举报在唯一活动键下聚合；处置、撤销、角色和信任管理都保存请求 ID、前后状态与操作者当时角色。

`v0.4.0` 的注册入口按 IP 和邮箱限制，Better Auth 的登录、重新验证、申请重置和执行重置使用 Redis 原子 Lua 计数。限流 key 中的邮箱、IP 和 Better Auth 标识先做 HMAC，不以明文写入 Redis。

反滥用策略要提供可解释错误码和重试时间，不能只返回模糊的 500。

## 7. 内容安全

- Markdown 渲染执行严格允许列表，禁止脚本、事件属性和危险协议。
- 用户提供的链接增加 `rel` 策略，外部内容不得突破 CSP。
- 上传先校验大小、签名和 MIME；图片解码与派生处理放在 Worker。
- 文件名不直接作为存储路径，下载响应设置安全的 Content-Type 和 Content-Disposition。
- 提及、链接预览和外部抓取必须防 SSRF，禁止访问本地、元数据和私有网段。
- 管理员查看被举报内容时同样执行安全渲染，不能认为后台天然可信。

`v0.7.0` 的实际实现以 Markdown 源为事实，预览和页面共用服务端清洗器；原始 HTML 不进入输出，危险协议和可能逃逸同源的路径被拒绝，站外图片不由服务器抓取或直接嵌入。上传按文件签名识别并核对 MIME，随机对象键不使用用户文件名；图片解码和 WebP 派生由 Worker 执行并限制 4000 万像素。附件媒体路由根据当前公开 Post 引用和上传者身份授权，返回 `nosniff` 与明确缓存策略；pending/failed 图片原件只以下载方式供上传者检查，不以内联图片执行。详细决策见 [ADR-0010](./adr/0010-replies-markdown-attachment-pipeline.md)。

## 8. Web 安全基线

- 所有改变状态的请求进行 CSRF 防护或使用严格同源机制。
- 建立 CSP、HSTS、`X-Content-Type-Options` 和合理的 Referrer Policy。
- 数据库查询参数化；原生 SQL 必须使用 Prisma 安全参数接口。
- 重定向目标只允许站内路径或显式白名单。
- OAuth 校验 state/nonce/PKCE（由具体协议和认证库负责）。
- 错误响应不包含堆栈、连接串、环境变量和用户隐私数据。
- 依赖更新通过自动检查进入合并请求，不能在生产启动时自动更新。

`v0.13.0` 将 CSP 脚本策略收紧为每请求随机 nonce 与 `strict-dynamic`，生产不允许 `script-src 'unsafe-inline'`；所有 NextBuf 自有写入 Route Handler 由自动扫描测试强制执行精确同源检查，Better Auth Handler 继续使用其原生 trusted origin 与 Cookie 防线。完整威胁模型和审计证据见[公开 Beta 就绪与加固记录](./16-public-beta-readiness.md)。

## 9. 管理后台安全

- 管理后台使用与前台相同的认证核心，但实施更严格的授权和会话策略。
- 修改角色、TL4、信任规则激活、站点设置、批量会话和审计导出属于高风险操作，需要固定确认文本和服务端再次授权。
- `v0.11.0` 通过 Better Auth 服务端 `verifyPassword` 为当前 Session 建立十分钟二次验证状态；不保存密码，不签发第二套 Cookie，撤销 Session 会同步清除验证状态。
- 仅 OAuth 管理员当前不能完成密码 step-up；正式公网运营前仍应评估 TOTP、Passkey/WebAuthn 或 OAuth step-up，并通过后续 ADR 扩展，不能绕过现有边界。
- 列表导出、批量操作和用户查询均需单独权限并记录审计。
- 普通用户和越权版主即使直接构造后台请求也必须由服务端返回 403/404；仅隐藏导航不是授权。
- 审计详情和 CSV 导出会递归移除 Password、Secret、Token、Cookie、Authorization、连接串和密文载荷字段；CSV 同时防止公式注入。
- “模拟用户”若未来实现，必须显著提示、限制敏感操作并完整审计。

## 10. 密钥与隐私

- `.env` 只提供示例，真实密钥不得提交仓库或写入镜像。
- 实例启动时检查会话密钥、加密主密钥和数据库密码的强度。
- 日志默认脱敏邮箱、令牌、Cookie、Authorization 和 Provider 密钥。
- 收集 IP、设备和行为数据必须有明确安全或运营目的及保留期。
- 备份应加密、限制访问并定期执行真实恢复演练。

## 11. 安全事件处理

至少准备以下流程：

1. 撤销会话、API Token 和受影响密钥。
2. 暂停相关 Provider 或高风险功能。
3. 保存日志和审计证据，避免边处理边覆盖。
4. 确认受影响用户、数据和时间窗口。
5. 修复、测试、发布并记录版本。
6. 按适用政策通知用户并复盘防止再次发生。
